# Decision Log

## 1. Integration test mock data corrected for per-tag-search pattern

- **Context**: The integration test `tests/integration/pipeline.test.ts` used single objects `{ id: N, name: 'xxx' }` for tag search responses in test 1, and a single array containing all 3 tags in one mock response for tests 2 and 3. The WordPress infrastructure implementation (validated by unit tests) sends one `GET /wp/v2/tags?search=xxx` request **per tag** and expects an array response (WP REST API collection format). With the original mocks, tests 2 and 3 would exhaust mock responses before all tag searches completed, causing `TypeError: Cannot read properties of undefined` on the third tag search.
- **Options considered**: (a) Normalize responses in `wordpress.ts` to handle both single-object and array formats from the search endpoint; (b) Fix the integration test mocks to match the per-tag-search contract established by the unit tests
- **Rationale**: Option (b) is correct. Adding format normalization in production code introduces a fallback/defensive pattern prohibited by the coding policy. The unit tests for `wordpress.ts` clearly specify arrays as the search response format (WP REST API standard). The integration test mock data was simply incorrect. Fixed: test 1 changed `{ id: 10 }` → `[{ id: 10 }]` and `{ id: 11 }` → `[{ id: 11 }]` for search responses; tests 2 and 3 expanded from 2 mock responses to 4 (3 per-tag searches + 1 post creation).

## 2. Non-null assertion (`!`) for WordPress config env vars in `generateAndPublish`

- **Context**: `generateAndPublish` loads `WP_SITE_URL`, `WP_USERNAME`, `WP_APP_PASSWORD` from `process.env`. TypeScript types these as `string | undefined`. The integration tests do not set these env vars, but stub `fetch` globally so the actual URL values never reach a real network call.
- **Options considered**: (a) Throw if env vars are missing (Fail Fast) — breaks integration tests since vars are unset; (b) Use `!` non-null assertion — TypeScript satisfied, runtime produces `'undefined/wp-json/...'` strings accepted by mocked `fetch`; (c) Create `.env.test` with dummy values and configure vitest to load it
- **Rationale**: Option (b) adopted. The CLI (`src/cli.ts`) calls `dotenv/config` before invoking the pipeline, making the vars available in production. For tests, the globally stubbed `fetch` accepts any URL string regardless of value. The `!` assertion is appropriate because the module contract assumes dotenv has already been loaded by the entry point. Option (c) would require modifying the vitest config or adding a test setup file, which is outside this task's scope.

## 3. Sequential tag resolution (for…of instead of Promise.all) in `publishPost`

- **Context**: Tag resolution requires one `GET` search per tag and optionally one `POST` create per missing tag. Using `Promise.all` executes all searches concurrently, making the order in which `vi.fn().mockResolvedValueOnce()` responses are consumed non-deterministic.
- **Options considered**: (a) `Promise.all` — concurrent execution, faster in production, but non-deterministic mock consumption breaks unit tests that rely on ordered `mockResolvedValueOnce` calls; (b) `for…of` sequential loop — deterministic order ensures each tag's search and optional create call consumes the correct mock response in sequence
- **Rationale**: Option (b). Unit tests are structured around an exact sequence of fetch calls (e.g., test 4: search→empty, create→id, search→found, post). Sequential processing guarantees this order. Performance difference is negligible for the typical use case of 2–10 tags per article.
